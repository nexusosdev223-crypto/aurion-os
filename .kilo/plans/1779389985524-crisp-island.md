# Plan: Supabase Unconfigured Guard — Full Fix & Test

## Diagnosis

`.env.local` has placeholder Supabase / Twilio API keys (`REPLACE_WITH_REAL_KEY`). Local `supabase-js` client rejects them at runtime → 500 errors on every API route that touches Supabase.

**Already partially fixed (mock-data guards added):**
- `src/app/api/ledger/route.ts` — `GET` returns mock ledger entries when Supabase is unconfigured.
- `src/lib/token.ts` — `calculateTokenVelocity()` returns mock velocity data when Supabase is unconfigured.

**Still broken (no guard — will 500):**

| File | Problem |
|---|---|
| `src/lib/supabase.ts` | Instantiates Supabase client unconditionally with placeholder keys — will throw on `import` if key is invalid. |
| `src/lib/compliance_gate.ts` | `getEconomicSnapshot()` queries Supabase directly; `auditGateDecision()` inserts via Supabase directly — no guard, blocks `/api/token/transfer` POST. |
| `src/app/api/token/transfer/route.ts` | Calls `compliance_gate` functions (which hit Supabase) and also does raw Supabase inserts — will 500 entire POST handler. |
| `src/app/api/metrics/velocity/executor.ts` | Background execution loop calls Supabase directly on every tick — throws and spams console after every 60 s. The import-and-run side-effect causes a Supabase call silently or with an internal try/catch that doesn’t surface success/failure. |

## Changes

### 1. `src/lib/supabase.ts`
Refactor to export a `createSupabase()` helper (guarded) and export `isValidSupabaseConfig()` for shared use:

```ts
import { createClient } from '@supabase/supabase-js';

export function isValidSupabaseConfig(): boolean {
  const url = process.env.NEXT_PUBLIC_SUPABASE_URL;
  const key = process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY;
  return !!(url && key && !key.includes('REPLACE_WITH_REAL'));
}

export function createSupabase() {
  if (!isValidSupabaseConfig()) return null;
  return createClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!
  );
}

export const supabase = createSupabase();
```

## Design Rationale

- **No breaking changes to real Supabase users** — when keys are valid, every guard short-circuits to the real Supabase path.
- **Single source of truth for guard** — `isValidSupabaseConfig()` exported from `supabase.ts`, imported everywhere; the check is explicit and easy to remove once real credentials are in place.
- **Executor is ground source** — executor.ts is the root that logic builds on, fixing it first aligns with the current workspace state from context earlier in the session.
- **One mock surface** — the apis return mock data, the UI renders correctly, integration tests pass.
- **Caching preserved** — existing `snapshotCache` and `velocityCache` patterns are kept; mock snapshots are cached so interactive UI still shows stable data.
- **Supabase client creation is guarded** — `createSupabase()` prevents getting a live client with placeholder keys.
